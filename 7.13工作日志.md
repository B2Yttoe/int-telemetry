# 7.13 工作日志

## 一、工作目标

本轮工作的核心目标是重新运行实验 2 中“原生 INT-MC 与增强 LEO-INT-MC”的对比实验，并修正此前实验中不够严格的预算对齐问题。

正式实验需要满足以下条件：

- 只使用 Iridium 66、Telesat 351 和 Starlink 1584 三种既定规模；
- 每种星座运行 48 个时间片；
- 原生组与增强组使用相同的 INT 观测预算；
- 增强组只能使用历史 Ground OAM 状态和可预测轨道信息，不能读取第一阶段隐藏真值；
- 同时比较遥测开销、直接覆盖率、节点状态误差和链路状态误差。

## 二、重跑前发现的问题

第一次正式重跑虽然完成了六条实验流水线，但验收时发现：原生组和增强组会分别根据各自的候选路径集合计算规模自适应预算。增强组加入 OAM 复测、拓扑复用和局部修复后，候选路径集合发生变化，因此两组得到的逐时间片硬预算并不完全相同。

这会造成一个实验公平性问题：即使两组都没有突破各自的预算，也不能严格声称它们是在“相同预算”下进行比较。

因此，本轮没有直接采用第一次重跑结果，而是将预算流程修改为：

1. 原生组根据覆盖目标生成 48 个时间片的预算日程；
2. 冻结原生组的 `scale_budget_bytes`；
3. 增强组逐时间片读取并复用这份预算日程；
4. OAM 复测、极区风险和局部修复只能改变预算内的探测优先级，不能提高预算上限。

## 三、代码修改

### 3.1 支持外部冻结预算日程

在 `stage2-int/tools/int-mc-path-selector.mjs` 中新增：

```text
--scale-budget-reference-csv <probe-summary-int-mc.csv>
```

选择器会从 CSV 中读取每个 `slice_index` 对应的 `scale_budget_bytes`，并执行以下检查：

- 时间片编号不能为空；
- 同一个时间片不能重复出现；
- 预算必须是大于 0 的有效数值；
- 参考文件必须覆盖本次实验的全部时间片。

增强组使用参考预算时，输出中的预算来源标记为：

```text
reference-schedule-hard-cap
```

### 3.2 明确硬预算来源

在 `stage2-int/tools/scale-adaptive-telemetry-budget.mjs` 中增加 `explicitBudgetSource`，用于区分：

- `explicit-hard-cap`：普通显式硬预算；
- `reference-schedule-hard-cap`：来自原生组冻结日程的硬预算；
- `coverage-derived-standard-metadata`：根据覆盖目标自动推导的预算。

### 3.3 实验脚本自动传递预算

在 `scripts/runExperiment2IntMcEnhancementComparison.mjs` 中完成以下调整：

- 原生组正常生成 `probe-summary-int-mc.csv`；
- 增强组自动使用原生组该文件作为预算参考；
- manifest 增加 `scale_budget_reference_csv`；
- manifest 和汇总 JSON 增加 `equal_scale_budget_schedule_enabled: true`；
- 报告中明确说明原生组与增强组均已重跑。

需要特别说明：启用规模自适应预算后，`max_paths_per_slice=12` 是兼容性的路径基准值，不是绝对路径上限。实验真正执行的硬约束是每个时间片的遥测字节预算。

### 3.4 回归测试

在 `scripts/testScaleAdaptiveBudgetPipeline.mjs` 中加入冻结预算测试：

- 先构造一份具有不同逐片预算的测试日程；
- 验证选择器输出的每片预算与参考日程完全一致；
- 验证预算来源为 `reference-schedule-hard-cap`；
- 验证实验 2 主脚本会把预算参考文件传递给增强组；
- 验证报告不会再声称“原生基线没有重跑”。

测试按照红绿循环执行：修改前测试得到 `20487 != 20000` 的预期失败，完成实现后测试通过。

## 五、预算公平性验收

三个星座的原生组和增强组均包含 48 个时间片。共检查 `3 × 48 = 144` 个增强时间片，结果如下：

| 星座 | 原生总预算上限/B | 增强总预算上限/B | 不匹配时间片 | 原生超限 | 增强超限 |
|---|---:|---:|---:|---:|---:|
| Iridium 66 | 728,666 | 728,666 | 0 | 0 | 0 |
| Telesat 351 | 4,716,757 | 4,716,757 | 0 | 0 | 0 |
| Starlink 1584 | 14,886,246 | 14,886,246 | 0 | 0 | 0 |

验收结论：本轮对比已经满足逐时间片相同硬预算，而不只是满足平均预算接近。

预算上限相同不代表实际发送字节必须相同。增强方法可以在同一上限内选择不同路径、压缩不同字段或保留更多高价值探测，因此还需要单独比较实际通信开销。

## 七、结果分析

### 7.1 已得到支持的结论

1. 在严格相同的逐时间片预算上限下，增强方案提高了三种规模的活动链路直接观测覆盖率。
2. CPU、队列、电量和全体链路利用率 MAE 在三种规模上均下降。
3. Starlink 1584 中，增强方案将实际遥测字节降低 21.10%，同时改善主要节点和链路重构误差，说明增强方案在大型动态拓扑上取得了明确的开销与精度联合优势。
4. Telesat 351 的实际字节只增加 0.63%，但 CPU、队列、电量和链路利用率误差明显下降，可以视为相近开销下的质量提升。
5. Iridium 66 的节点模式准确率由 92.68% 提高到 93.78%，链路状态准确率由 83.08% 提高到 87.40%。

### 7.2 必须保留的适用边界

Iridium 66 中，仅针对未直接观测链路计算的利用率补全 MAE 从 9.5491 上升到 10.8570，退化 13.70%。与此同时，全体链路利用率 MAE仍下降 13.22%。这说明增强方案在小型星座上通过增加直接观测改善了整体状态恢复，但剩余未观测样本更可能集中在难以补全的位置。

因此，本轮不能宣称“增强方案在所有星座、所有指标和所有开销上全面优于原生 INT-MC”。更严谨的结论是：

> 增强 LEO-INT-MC 在中大型动态 LEO 星座上表现出明确的等预算优势；在小型星座上能够改善多数节点和链路指标，但仅推断利用率误差仍存在适用边界。

## 八、无真值泄漏验收

三种规模均满足：

- `observability_mode = oam-only`；
- `hidden_stage1_metrics_available_to_planner = false`；
- 动态状态来源为 `lagged-ground-oam-estimates`；
- OAM 反馈滞后一个时间片；
- 因果反馈违规数量为 0；
- 合并反馈中使用真值误差的记录数量为 0；
- 第一阶段真值只在实验结束后的 MAE 和准确率评估中使用。
